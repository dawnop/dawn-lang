# 序 6 调研：`Int`/`char` 值类型特化 + `String` 非物化字符访问

> 动码前的**调研与方案**，不是设计定稿。结论：**retro 对序 6 的描述与代码现状有实质出入**，
> 照原样做会推翻两条深思熟虑过的决策、且**对主要痛点收效甚微**。建议拆开做，先做便宜的那半。

## 一、结论先行

| | retro 的说法（`m6-retro.md:61-74`、`:261`） | 代码与实测的现状 |
|---|---|---|
| `Int` 装箱 | 「泛型全擦除 + 装箱」，暗示 Int 到处是堆对象 | **局部变量/参数/返回值/具体记录字段/闭包捕获全是 `J`（原生 long），零装箱**。装箱只发生在**声明位置是类型变量**的地方 |
| `char = Int` | 列为缺陷「把 char 变成堆上的 Int」 | **`char` 根本不是堆对象**——它就是 `Int`，即原生 `long`。Go 的 rune 模型，spec §1.5 明写、`CharTest.kt` 钉住，是**刻意设计** |
| 「一篇文章正文 = 百万个装箱 Int」 | 主要机制 | **对 ASCII 不成立**（`Long.valueOf` 缓存 −128..127）。实测装箱只占 **+40%**；大头是 `ArrayList` 物化本身 |
| 修法 | 「值类型特化」 | **对 ASCII 一分钱不省**：`long[]` 的 8 字节 == 现在的指针 8 字节。真正省的是**不物化** |

**建议**：把序 6 拆成两件事，只做 **A（String 非物化访问，库层）**，把 **D（单态化/值类型特化，编译器大改）
无限期搁置**，除非基准证明 A 之后还有不可接受的缺口。**B（元组特化）**便宜且独立，可顺手做。

## 二、现状（三份调研的交集，均有 file:line 佐证）

### 2.1 装箱策略就是一行

`CodeGen.kt:645` —— `adaptTo(actual, declared) = if (declared is TVar) box(actual)`。
**值本身是不是具体类型无关紧要，只看声明位置。** 于是：

| 位置 | `Int` 的描述符 | 装箱 |
|---|---|---|
| 局部变量 / 参数 / 返回值 | `J`（LLOAD/LSTORE） | **否** |
| **具体**记录/ADT 字段 | `J` | **否** |
| 闭包捕获 | `J` | **否** |
| `List[Int]` 元素、Map 键值 | `Object` | 是 |
| **元组分量（永远）** | `Object` | 是 |
| `Option[Int]`/`Result` 载荷 | `Object` | 是 |
| 泛型函数实参、trait 字典调用、`Fn1.apply` | `Object` | 是 |

即：**剩下的装箱恰好等于「泛型位置的装箱」，而那正是 D3 与 D9 明确选择的东西**
（D3「泛型走独立的、无聊的参数多态（擦除 + 装箱）」；D9「不做单态化——native-image
代码膨胀 + 编译时间不划算」）。D6 还专门写着「为想象中的需求预付架构成本是小项目最常见的死法」。
**序 6 原样做 = 推翻 D3 + D9，并顶着 D6。** 门槛必须高。

### 2.2 真正的缺口是 `String` 没有 `char_at`

`Bytes` 有 `byte_at`/`byte_slice`/`byte_index_of`（`Types.kt:454-467`），**`String` 一个都没有**。
要读一个字符只有两条路：`substring(s,i,i+1)`（返回 `String` 不是 `Int`），或把整个串炸成 `List[Int]`。

**这个不对称就是全部痛点的来源**，而且**序 4（Bytes）已经用同样的手法解决过一次**：
retro 引用的 `multipart.dawn` 那句警告（"never code_points on the content ... would box millions of Ints"）
**现在已经不存在了**——`f9c339e` 加了 `byte_index_of`/`byte_slice` 之后，multipart 改成在 Bytes 上定位结构，
**没动类型系统**。序 6 的 String 那半，是同一个剧本的重演。

### 2.3 受害现场（按输入规模分三档）

- **档 1（真疼，整篇正文）**：`json/parser.dawn:126` `parse` 把**整个 JSON 文档**炸成 `List[Int]` 才开始词法分析
  （**retro 里根本没提，却是最热的路径**——`POST /api/articles` 的正文在 JSON 里，要付两遍）；
  `repo_article.dawn:14` `word_count`（且在 `:72` 按行调用 → `GET /api/articles?size=10` 物化**十篇正文**只为算个整数）；
  `search.dawn` 的 `strip_md`/`excerpt_html` 链。
- **档 2（小输入但每字符分配一个 List）**：`export.dawn:22,33`、`search.dawn:158`。
- **档 3（装箱无关，纯粹是没有 `char_at`）**：`needs_quote`、`slugify`、`only_slug_chars`、
  以及所有 `from_code_points([13,10])` 拼 CRLF。**这一档只要 API 补齐就消失，与性能无关。**

顺带两个独立的人体工学缺口：`index_of` **没有 `from` 参数**（逼出 `search.dawn:166` 手写版）、
`replace`/`slice` **不是内建**（每个应用各手写一遍）。

## 三、实测（本轮亲自跑，非推测）

基准载体：`dawn run examples/m4/json <file>`，纯 Dawn、多模块、正打在 `code_points` 上。

**1) 物化的代价是数量级的**：1.3MB JSON → **peak RSS 467–500MB**（≈ **360× 输入**）。

**2) 装箱只是其中一小部分**——两份语料**码点数完全相同**（902901），只差码点值：

| 语料 | peak RSS（3 次） | 机制 |
|---|---|---|
| 正文全 ASCII（码点 < 128） | 453 / 478 / 583 MB | **命中 `Long` 缓存，一个 Long 都没分配** |
| 正文全中文（码点 > 19967） | 633 / 744 / 727 MB | 每字符新分配一个 16 字节 `Long` |

**读法**：装箱带来 **+40%**；但**纯 ASCII 一个 Long 都没装箱，仍然烧掉 453MB**。
大头是 `ArrayList` 物化本身——90 万个元素的指针骨架 + 扩容倍增拷贝 + GC 余量。

**3) 这一条直接判了「值类型特化」的死刑**：把 `List[Int]` 特化成 `long[]` 后，每码点 **8 字节**——
**和现在的指针一样是 8 字节**。也就是说**特化对 ASCII 场景省下的是零**，对中文省掉那 16 字节的 Long。
而**不物化**省掉的是**全部**。retro 把因果归错了：痛的不是"装箱"，是"物化"。

## 四、方案

### 方案 A —— `String` 非物化字符访问（**推荐**，库层，不动类型系统）

补齐 String 侧的 `Bytes` 对等物，让档 1/2/3 的调用点都不再需要 `code_points`。与 D3/D6/D9 全部相容，
是序 4 已验证过的形状。

**待定的设计题（A 的真正难点，不是工作量而是 API 形状）**：Java `String` 是 UTF-16，
而 Dawn 现有的 `str_len`/`substring`/`index_of` **全部按码点索引**（`CodeGen.kt:2104-2113` 做 UTF-16↔码点换算）。
于是「按码点索引的 `char_at(s, i)`」是 **O(n)**，循环里用就是 **O(n²)**——比现在还糟。三个候选：

- **A1 游标**：`next_char(s, cursor) -> Option[(Int, Int)]`（码点 + 下一个游标）。游标对用户不透明，
  实为 UTF-16 下标。O(1) 每步，能满足 JSON 词法/回溯。**代价**：返回元组 → 撞上元组装箱（见方案 B）。
- **A2 回调遍历**：`each_char(s, f)` / `chars_fold(s, init, f)`。最简单、O(n)、零物化，
  但**不支持回溯**，JSON 词法器用不了。
- **A3 暴露 UTF-16**：`char_at_utf16` + `utf16_len`。O(1)、无元组，但把编码模型漏进语言，
  与现有全码点索引的 API 精神冲突。

倾向 **A1**（唯一能覆盖档 1 里 JSON 词法器的），且它给方案 B 提供了独立的动机。
**顺手补**：`index_of(s, sub, from)`、`replace`、`slice` —— 消掉三处手写实现。

**收益**：预计把 JSON 基准的 467MB 打到接近输入规模。**成本**：中。**风险**：低（纯新增，旧 API 不动）。

### 方案 B —— 元组特化（便宜、独立、修一个真不一致）

`genTupleClass(n)` **只按元数生成类**，字段硬编码 `Object`（`CodeGen.kt:438`）——它连类型都拿不到。
而 `genCtorClass` 用 `descOf(f.type)`（`:701`），**具体记录的字段是全原生的**。
于是 `type P = { x: Int, y: Int }` 零装箱，`(Int, Int)` 两个装箱——**同样的类型，两种待遇**。

改法：元组类按**类型形状**生成（`Tuple2$JJ`），镜像 `genCtorClass` 的现成做法。类数量受实际用到的形状数约束。
**如果走 A1，这条是前置**（游标返回元组，不特化就白省）。**成本**：低-中。**风险**：低。

### 方案 C —— 窄化的容器/字典特化

如 `Ord[Int].cmp` 现在每次比较要拆两个 Long 箱（`CodeGen.kt:979`，底下其实是条 `LCMP`）；
或给 `map`/`fold` 加原生特化。**收益零散、且 A 之后大概率不再是瓶颈**。**建议：不做，等基准说话。**

### 方案 D —— 真·值类型特化 / 单态化

推翻 D3 + D9，顶着 D6。**且第三节的实测显示它对 ASCII 场景收益为零**。
**建议：无限期搁置**，除非 A+B 落地后基准仍显示不可接受的缺口——那时再谈，也有基准可依。

## 五、建议路线

1. **第 0 步（前置，retro 自己要求的"性能基准护栏"）**：`scripts/bench.sh` —— `dawn run examples/m4/json`
   跑生成的语料，`/usr/bin/time` 取 **peak RSS 中位数**（wall 噪声大：0.48–0.71s；RSS 是 460MB 量级，信噪比极高）。
   录基线、超阈值报警。约半天。**没有它，A 的收益和 D 的必要性都只是嘴上说。**
2. **A1 定稿**（游标 API 的形状是唯一真设计题）→ 落地 A + 顺手补 `index_of(from)`/`replace`/`slice`。
3. **B（元组特化）**——若 A1 定为游标则它是前置，否则可独立择机。
4. **重测**。若缺口已可接受 → 序 6 **结案，D 记入"被实测推翻的方案"**，与 `@NonNull`、`List[T]→T[]` 桥并列。
